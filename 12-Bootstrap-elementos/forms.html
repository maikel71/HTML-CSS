Forms - Formularios

Overview - Visión general

Ejemplos y directrices de uso de estilos de control de formularios, opciones de diseño y componentes personalizados para crear una amplia variedad de formularios.

Los controles de formulario de Bootstrap amplían nuestros estilos de formulario Rebooted con clases. Utilice estas clases para optar por sus visualizaciones personalizadas para una representación más coherente en todos los navegadores y dispositivos.

Asegúrese de utilizar un atributo de type adecuado en todas las entradas (por ejemplo, email para la dirección de correo electrónico o number para la información numérica) para aprovechar los nuevos controles de entrada como la verificación de correo electrónico, la selección de números, etc.

He aquí un ejemplo rápido para demostrar los estilos de formulario de Bootstrap. Sigue leyendo para obtener documentación sobre las clases requeridas, el diseño del formulario y mucho más.

Disabled forms - formularios deshabilitados

Añade el atributo booleano disabled en una entrada para evitar interacciones del usuario y hacer que aparezca más clara.


Añada el atributo disabled a un fieldset para desactivar todos los controles que contenga. Los navegadores tratan todos los controles de formulario nativos (elementos input, select y button) dentro de un fieldset disabled como desactivados, lo que impide las interacciones con el teclado y el ratón sobre ellos.

Sin embargo, si el formulario también incluye elementos tipo botón personalizados como (a) class="btn btn-*"..."(/a), sólo se les asignará el estilo pointer-events: none, lo que significa que aún se pueden enfocar y utilizar con el teclado. En este caso, debe modificar manualmente estos controles añadiendo tabindex="-1" para evitar que reciban el focus y aria-disabled="disabled" para indicar su estado a las tecnologías de asistencia.

Accessibility - Accesibilidad

Asegúrese de que todos los controles del formulario tengan un nombre accesible adecuado para que los usuarios de tecnologías de asistencia puedan conocer su finalidad. La forma más sencilla de conseguirlo es utilizar un elemento label o, en el caso de los botones, incluir un texto suficientemente descriptivo como parte del contenido de (button)...(/button).

Para situaciones en las que no es posible incluir una (label) visible o un contenido de texto adecuado, existen formas alternativas de seguir proporcionando un nombre accesible, como por ejemplo:

  • Elementos (label) ocultos mediante la clase .visually-hidden
  • Señalar un elemento existente que puede actuar como etiqueta utilizando aria-labelledby
  • Proporcionar un atributo de title
  • Establecer explícitamente el nombre accesible de un elemento mediante aria-label

Si ninguno de ellos está presente, las tecnologías de asistencia pueden recurrir al atributo de placeholder como alternativa al nombre accesible en los elementos (input) y (textarea). En los ejemplos de esta sección se sugieren algunos enfoques específicos para cada caso.

Aunque el uso de contenido visualmente oculto (.visually-hidden, aria-label e incluso contenido de placeholder, que desaparece una vez que el campo de formulario tiene contenido) beneficiará a los usuarios de tecnología de asistencia, la falta de texto de etiqueta visible puede seguir siendo problemática para determinados usuarios. Por lo general, lo mejor es que haya algún tipo de etiqueta visible, tanto por motivos de accesibilidad como de usabilidad.

Form text - Texto del formulario

El texto de formulario a nivel de bloque o a nivel de línea puede crearse utilizando .form-text.

El texto de formulario debe asociarse explícitamente con el control de formulario al que se refiere mediante el atributo aria-describedby. Esto garantizará que las tecnologías de asistencia -como los lectores de pantalla- anuncien este texto de formulario cuando el usuario enfoque o entre en el control.


El texto del formulario debajo de las entradas puede estilizarse con .form-text. Si se utiliza un elemento a nivel de bloque, se añade un margen superior para facilitar el espaciado con respecto a las entradas anteriores.

Su contraseña debe tener entre 8 y 20 caracteres, contener letras y números, y no debe contener espacios, caracteres especiales ni emoji.

El texto en línea puede utilizar cualquier elemento HTML en línea típico (ya sea un (span), (small) o cualquier otro) con nada más que la clase .form-text.

Checks & radios

Cree casillas de verificación y radios coherentes entre navegadores y dispositivos con nuestro componente de verificación completamente reescrito.

Approach - Acercarse

Las casillas de verificación y los radios predeterminados del navegador se sustituyen con la ayuda de .form-check, una serie de clases para ambos tipos de entrada que mejora el diseño y el comportamiento de sus elementos HTML, que proporcionan una mayor personalización y coherencia entre navegadores. Las casillas de verificación sirven para seleccionar una o varias opciones de una lista, mientras que los radios sirven para seleccionar una opción entre varias.

Estructuralmente, nuestros (input)s y (label)s son elementos hermanos a diferencia de un (input) dentro de un (label). Esto es un poco más verboso, ya que debe especificar los atributos id y for para relacionar los elementos (input) y (label). Utilizamos el selector de hermanos (~) para todos nuestros estados (input), como :checked o :disabled. Cuando se combina con la clase .form-check-label, podemos estilizar fácilmente el texto de cada elemento basándonos en el estado del (input).

Nuestras comprobaciones utilizan iconos personalizados de Bootstrap para indicar los estados comprobado o indeterminado.

Checkbox toggle buttons - Botones de casilla de verificación



Visualmente, estos botones de alternancia de casilla de verificación son idénticos a los botones de alternancia del complemento de botón. Sin embargo, las tecnologías de asistencia los transmiten de manera diferente: los lectores de pantalla anunciarán las casillas de verificación como "marcadas"/"no marcadas" (ya que, a pesar de su apariencia, siguen siendo fundamentalmente casillas de verificación), mientras que los botones de alternancia del complemento de botón serán anunciado como “botón”/“botón presionado”. La elección entre estos dos enfoques dependerá del tipo de alternancia que esté creando y de si la alternancia tendrá o no sentido para los usuarios cuando se anuncie como una casilla de verificación o como un botón real.

Forms - Formularios

Cada grupo de campos de formulario debe residir en un elemento (form). Bootstrap no proporciona ningún estilo predeterminado para el elemento (form), pero hay algunas potentes características del navegador que se proporcionan de forma predeterminada.

  • ¿Eres nuevo en los formularios del navegador? Consulte la documentación sobre formularios de MDN para obtener una visión general y una lista completa de los atributos disponibles.
  • Los (button)s dentro de un (form) tienen por defecto type="submit", así que esfuérzate por ser específico e incluye siempre un type.

Dado que Bootstrap aplica display: block y width: 100% a casi todos nuestros controles de formulario, los formularios se apilarán verticalmente por defecto. Se pueden utilizar clases adicionales para variar este diseño en cada formulario.

Las utilidades de margen son la forma más sencilla de añadir estructura a los formularios. Recomendamos ceñirse a las utilidades de margin-bottom y utilizar una única dirección en todo el formulario para mantener la coherencia.

Siéntete libre de construir tus formularios como quieras, con (fieldset)s, (div)s, o casi cualquier otro elemento.

Validation

Proporcione información valiosa y procesable a sus usuarios con la validación de formularios HTML5, mediante comportamientos predeterminados del navegador o estilos personalizados y JavaScript.

Somos conscientes de que actualmente los estilos de validación personalizados del lado del cliente y la información sobre herramientas no son accesibles, ya que no están expuestos a las tecnologías de asistencia. Mientras trabajamos en una solución, te recomendamos que utilices la opción del servidor o el método de validación por defecto del navegador.


How it works - Cómo funciona

Así es como funciona la validación de formularios con Bootstrap:

  • La validación de formularios HTML se aplica mediante dos pseudoclases de CSS, :invalid y :valid. Se aplica a los elementos (input), (select) y (textarea).
  • Bootstrap asigna los estilos :invalid y :valid a la clase padre .was-validated, normalmente aplicada al (form). De lo contrario, cualquier campo obligatorio sin valor se mostrará como inválido al cargar la página. De esta forma, puedes elegir cuándo activarlos (normalmente después de que se intente enviar el formulario).
  • Para restablecer el aspecto del formulario (por ejemplo, en el caso de envíos dinámicos de formularios mediante Ajax), vuelva a eliminar la clase .was-validated del (form) después del envío.
  • Como alternativa, se pueden utilizar las clases .is-invalid y .is-valid en lugar de las pseudoclases para la validación en el servidor. No requieren una clase padre .was-validated.
  • Debido a limitaciones en el funcionamiento de CSS, no podemos (en la actualidad) aplicar estilos a una (label) que precede a un control de formulario en el DOM sin la ayuda de JavaScript personalizado.
  • Todos los navegadores modernos soportan la API de validación de restricciones, una serie de métodos JavaScript para validar controles de formularios.
  • Los mensajes de respuesta pueden utilizar los valores predeterminados del navegador (diferentes para cada navegador, y no estilizables mediante CSS) o nuestros estilos de respuesta personalizados con HTML y CSS adicionales.
  • Puede proporcionar mensajes de validez personalizados con setCustomValidity en JavaScript.

    • Teniendo esto en cuenta, considere las siguientes demostraciones de nuestros estilos de validación de formularios personalizados, clases opcionales del lado del servidor y valores predeterminados del navegador.

Custom styles - Estilos personalizados

Para los mensajes personalizados de validación de formularios de Bootstrap, tendrás que añadir el atributo booleano novalidate a tu (form). Esto desactiva los tooltips de retroalimentación por defecto del navegador, pero todavía proporciona acceso a las APIs de validación de formularios en JavaScript. Intente enviar el siguiente formulario; nuestro JavaScript interceptará el botón de envío y le enviará los comentarios. Al intentar enviar, verá los estilos :invalid y :valid aplicados a los controles del formulario.

Los estilos de retroalimentación personalizados aplican colores, bordes, estilos de enfoque e iconos de fondo personalizados para comunicar mejor la retroalimentación. Los iconos de fondo para (select) sólo están disponibles con .form-select, y no con .form-control.

  • (input)s y (textarea)s con .form-control (incluido hasta un .form-control en grupos de entrada)
  • (select)s con .form-select
  • .form-checks

Tooltips - Información sobre herramientas

Si el diseño de tu formulario lo permite, puedes cambiar las clases .{valid|invalid}-feedback por las clases .{valid|invalid}-tooltip para mostrar la información de validación en un tooltip con estilo. Asegúrate de tener un padre con position: relative en él para el posicionamiento del tooltip. En el ejemplo de abajo, nuestras clases de columna ya tienen esto, pero su proyecto puede requerir una configuración alternativa.
